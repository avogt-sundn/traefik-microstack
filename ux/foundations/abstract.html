Sachbearbeiter Command Dashboard — Design Foundations

ux/foundations/ · IHK Carnet Administration · May 2026
This document synthesises the design foundations for the Sachbearbeiter Command Dashboard: the single-screen model, the clerk's mental model, the command field metaphor, the chat-stream history, the reset model, the instant-resolution performance contract, and the mitigations for known objections. It is a shareable reference for engineers, designers, and stakeholders who need to understand the "why" before touching the "what."

1The Problem: Re-orientation Is the Real Tax

A Sachbearbeiter in IHK Carnet administration handles 60–100 Vorgänge per day across heterogeneous entity types — Firmen, Carnets, Prüfungen — in multiple Kammerbezirke. Traditional enterprise UIs (SAP GUI, ServiceNow, Oracle Forms) force clerks to navigate between screens. The cost is not the click. It is the re-orientation after arrival: every screen transition requires the clerk to answer "Where am I? What can I do here? How do I get back?" At high volume, this compounds into hours of lost productivity per week.

The design goal is zero re-orientation tax: a fixed three-zone layout whose spatial stability answers orientation questions through peripheral vision, without any active reading.

Legacy systemPatternRe-orientation cost
SAP GUI7400+ transaction codes → separate screensClerks memorise paths instead of doing work. Months of training.
ServiceNowList → Detail → Related list → Back3–5 page loads per ticket. Queue awareness lost during detail work.
Oracle FormsMenu → Form → Tab → Sub-form → Save → Back4–6 navigation steps per CRUD operation.

2Three Zones, Zero Navigation

The dashboard is one page with three vertically stacked zones. Each zone answers one cognitive question that is stable across all tasks, all entity types, and all experience levels. The zones never move, never hide, and never change layout.

ZoneCognitive questionStability guarantee
Zone 1 — Scope bar"Who am I? What can I see?"Never scrolls, never hides, never changes layout
Zone 3 — Chat history"What happened? What is waiting?"Always above Zone 2; scrolls upward; bottom = newest
Zone 2 — Command field"What am I doing right now?"Anchored at the bottom, always same position, same interaction model

3The Clerk's Mental Model

The top layer of any administrative system is the mental model the clerk brings to the desk. A Sachbearbeiter thinks in Vorgänge, not in database rows. They think in permitted actions, not in CRUD operations. A system that exposes its data model to the clerk has failed before the first interaction.

ConceptMeaning to the clerkWrong system equivalent
VorgangA case with a history, a current state, and pending actionsA database row
ZuständigkeitWhat I am responsible for and permitted to act onA permission bitmask
ArbeitsschrittA discrete action that advances a Vorgang and leaves a traceA form submit

Every screen, every search result, every autocomplete suggestion must be expressible in these three terms. The system must speak the clerk's vocabulary back to them.

The Command Field as Interpreter

The central command field is not a search box. It is the clerk's command line — a direct expression of intent over the domain object graph. Each token narrows the entity set and updates the available action set simultaneously. The clerk never sees a results page; they see a continuously narrowing intention.

Keystroke sequence          What the system shows
──────────────────          ──────────────────────────────────────────────────
"g"                    →    Günther Maschinenbau AG  ·  Gauss Metallbau GmbH
"gün"                  →    Günther Maschinenbau AG                          [entity resolved]
"gün car"              →    Günther Maschinenbau AG  ›  Carnet #2024-00456   [relation followed]
"gün car ver"          →    … ACTION: Zu anderer Firma verschieben ↵         [method matched]
"gün car ver mül"      →    … verschieben → Müller Werkzeugbau GmbH          [parameter resolved]
[ENTER]                →    Arbeitsschritt ausgeführt · Zeitstrahl aktualisiert
Authorization rule

Actions are filtered at query time, not render time. A clerk in Kammer Aachen never sees a greyed-out "Limit ändern" button. They do not see the button at all. The UI is a permission-scoped workspace, not a permission display system.

4Der Strom — Chat as History

Zone 3 is the history of the clerk's session and all events on entities within their scope. Its spatial metaphor is the chat transcript: the most deeply trained spatial model of the 2020s workforce — everyone who uses a phone already knows it. The command field is the message input, anchored at the bottom. History flows upward above it. The most recent exchange is directly above the input.

Two speakers, two visual alignments:

  • Sie: (left-aligned) — the clerk's typed command. What I said.
  • Aktion: (right-aligned) — the system's response. What happened.

A colleague event or system automation appears as an Aktion: bubble without a preceding Sie: — exactly like receiving a message in a group chat. The clerk sees what happened while they were away by scanning to the bottom of the stream.

Vorgangshistorie — the Entity Kassenbuch

The chat stream shows the clerk's personal session. But a Vorgang has a lifecycle wider than any one clerk: customer portal submissions, colleague actions, external partner responses, system automations. This full history lives in the Vorgangshistorie — a sidebar panel that opens when the clerk selects a specific entity.

ViewQuestion answeredScope
Chat stream (Zone 3)"What did I do?"My session, all entities
Vorgangshistorie panel"What happened to this entity?"One entity, full lifecycle, all actors

Both views share the Kassenbuch contract: append-only, sequential, attributed, irreversible. The panel opens as a sidebar; it overlays the left portion of the stream but does not hide the command field. It is not a modal and not a separate page.

5The Reset Model — git reset, Not Ctrl+Z

The Arbeitsstrahl is the clerk's personal command trail. Traditional undo (Ctrl+Z) processes in strict reverse order: undoing action 5 to fix action 3 also destroys action 4. The Arbeitsstrahl uses a different model: git reset with automatic causal dependency analysis.

The clerk clicks any prior "Sie:" bubble and the system evaluates which subsequent actions depended on it (same entity, or output-as-input) and which were independent. Independent actions survive the reset automatically. Only causally dependent actions are reversed. The reset itself is a Kassenbuch entry — nothing is silently erased.

Traditional Ctrl+ZArbeitsstrahl reset
Strict reverse order — must undo 5, 4, 3 to fix 3Direct jump to any entry — independent work preserved
No preview of what will be undoneConfirmation panel shows exactly which actions are dependent
Silent — no record that undo happenedAuditable — the reset is itself a Kassenbuch entry
Destroys intermediate stateUndone entries preserved in reflog for compliance

6Instant Resolution — The Performance Contract

The command field's promise is every keystroke produces a visible preview within ~50 ms, regardless of backend round-trip time. This is achievable only if the client never fires a request on keydown. All visible suggestions come from data that arrived before the user started typing. The backend's job is to anticipate what the user will need next and ship it ahead of time.

TriggerStrategyMechanism
Session startGlobal entity indexCompact in-memory index of all Firmen/Carnets/Prüfungen — all token matching runs locally, zero network latency per keystroke
User typing, top match stable >150 msSpeculative prefetchBegin fetching top candidate's anchor payload before Space is pressed; cancel if match changes
Entity token locked (Space)Warm cacheFetch all actions, related objects, and parameter candidates in one round-trip — client is ready before the user types the action token
Zone 3 / Offene Vorgänge renderedViewport prefetchBatch-fetch anchor payloads for all visible entities; clicking any row pre-fills an already-cached token
Confirmation panel must appearCo-packaged payloadConfirmation text and validation results are inside the anchor payload — no second request needed
Rule

Do not fire a request on every keydown. Debounce is not sufficient — the goal is zero keydown requests. The network only moves when the entity resolves (Space), not while the user is typing.

7Known Objections and Their Mitigations

A traditional web application designer raises eight structural objections to the single-screen model. The table below classifies each by whether it is fully mitigated within the single-screen contract, partially mitigated with design additions, or a genuine tension to be acknowledged.

Fully mitigated — URL contract

The rationale conflates "page navigation" with "context loss." They are separate concerns. The single-screen contract does not require zero URLs. history.pushState() on every significant state transition (entity focused, panel opened, form section reached) gives the URL all its power — shareability, browser history, back-button, test fixtures — without any navigation event or context loss. The screen never unloads.

Counterargument costMitigation
Clerk cannot share a Vorgang URL with a colleaguepushState on entity focus; copy-URL button on entity header chip
Back button throws clerk to loginIntercept popstate; treat it as state restore inside the screen, not navigation
Supervisor cannot audit from browser historyFollows from pushState — entity-level transitions appear in history
Support cannot reproduce a bug from a URLFollows from stable URL scheme per entity type and state
Automated tests cannot target a specific entityNavigate to /vorgang/2024-00456; screen restores to that state
Full-width panel has no URLpushState on panel open: /carnet/2024-00456/warenpositionen; closing restores prior URL

Partially mitigated — design additions required

Counterargument costMitigation
Novice doesn't know to type "carnet" to discover Carnets existEmpty command field shows category chips (Carnets, Firmen, Prüfungen) as mouse-clickable affordances that pre-fill the token — zero vocabulary required
Zone 3: is this entry audit log or private session history?Visual distinction: Sie: bubbles are typographically lighter (italic, lower opacity); hover reveals "Privat – nicht im Protokoll." Aktion: bubbles carry source attribution (K. Huber, T. Klein, [System], [Portal])
BITV 2.0 accessibility: screen readers cannot parse the command fieldImplement as <input role="combobox" aria-autocomplete="list" aria-expanded aria-owns="suggestion-list">; zones map to <header>, <main>, <aside> landmarks — BITV compliance is a first-class design constraint, not an afterthought

Genuine tensions — acknowledged, not resolved

TensionHonest position
Non-touch-typists are slower with tokens than with click targetsThe mouse-navigable suggestion list and category chips reduce the gap, but the token model still favours typists. Accepted tradeoff for high-volume professional use.
No user research with actual IHK Carnet clerksAll citations are theoretical. The design must be validated with 5 representative clerks before any commitment to implementation. This is an open empirical question.
Full-width panel (Zone 3 consumed) is nominally "one screen"Zone 1 (scope) remains visible; the contract holds there. Zone 3 is pushed below the viewport. The honest position: the single-screen claim applies to Zone 1 always visible and ESC always exits, not to simultaneous visibility of all three zones under maximum complexity.

8What This Rules Out

The design decision has consequences that must be accepted, not negotiated:

  • No detail pages as separate routes. All detail is shown inline in Zone 2 (command resolution + inline panels) or Zone 3 (stream entries + Vorgangshistorie sidebar).
  • No settings page. Scope is set in Zone 1. Preferences are a command (einstellungen sprache deutsch).
  • No onboarding wizard. A new clerk starts with the same screen as a veteran. The command field's category chips and suggestion mechanism are the onboarding.
  • No modal confirmation dialogs. The completion preview line in Zone 2 is the confirmation. When the preview shows exactly what will happen, there is nothing left for a modal to add.
  • No dashboard widgets or KPI tiles. The Offene Vorgänge sidebar is the only "dashboard" content — and it is actionable, not decorative.
  • No multi-step wizard flows. Complex forms use progressive inline expansion inside Zone 2, sectioned with TAB / SHIFT+TAB, with auto-save drafts appearing in Offene Vorgänge for interrupted sessions.

9Recommended Build Sequence

The design has a strict dependency order. Building the UI before the domain model produces a UI that encodes the wrong abstractions.

  1. Name the clerk's three concepts — Vorgang, Zuständigkeit, Arbeitsschritt — and map them to domain entities. No UI work until this is stable.
  2. Define the uniform entity interface: resolve(), actions(user, perms), synonyms, history(). Every entity type implements the same contract.
  3. Design the command interpreter as a domain service — not a UI component. The suggestion list is the type system made visible.
  4. Wire authorisation into query time. The API returns only what the clerk may see and do. Never filter at render time.
  5. Build the timeline as an append-only ledger at the data layer, with source attribution on every entry.
  6. Build the UI last — three zones that render the domain model, nothing more.
  7. Validate with five actual IHK Carnet clerks before committing to production implementation.
